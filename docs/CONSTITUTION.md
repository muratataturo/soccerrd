# Constitution — 持続可能なアプリケーション開発のための哲学

> このドキュメントは soccerrd における **最上位の判断基準** である。
> 個別の戦術・実装判断は本憲法と整合しなければならない。整合しない場合は、まず本憲法を更新する。

---

## 第1条 プロダクトが目指す方向性

### 1.1 ターゲットとコアユース
- **誰に**: ユース年代・アマチュアでサッカーをプレイする選手・コーチ・監督
- **何を**: サッカーの思考と言語化を支援する
- **どのように**: 利用すればするほど自動的にサッカー解析と示唆が「**サッカーNote**」として蓄積され、その Memory から
  - **動的に検索可能**
  - **横断的な組織の知識**

  として常に還元される

### 1.2 進化の構造（フライホイール）

```
利用 → 思考と言語化の支援 → サッカーNote の蓄積 → 動的検索・組織知化 → 還元 → さらなる利用
```

このフライホイールが回り続ける機能のみが、プロダクトの方向性に整合する。

---

## 第2条 プロダクトを作成していく方向性

機能開発は次の 2 軸のいずれか（または両方）を満たさなければならない。

### 2.1 想定ペルソナにおける UX 差分・体験向上
- 「この機能だと、この地点での離脱が高まる」
- 「この機能が不足しているため、プロダクト哲学を体現できていない」

ペルソナ別の UX ジャーニーは [product/](./product/) に蓄積する。

### 2.2 競合劣位の解消
- 現状の初期アプリケーションを中心に、似ているアプリケーション・プロダクトを **その時点における web 検索を通じて調査する**
- 競合観察は [research/competitors/](./research/competitors/) に時系列で蓄積する

---

## 第3条 優先度付けの基準

### 3.1 売上ドリブン（基本原則）
段階により変化するが、基本的には **売上を上げる方向** の機能開発を優先する。
- **受注（acquisition / conversion）**
- **チャーン回避（retention）**

### 3.2 スコープの確認義務
新機能・改善案を扱う際、Claude は **どちらの方向（受注 / チャーン回避）にスコープを切るかを常にユーザに聞き返す**。
方向が定まらないまま実装に進んではならない。

詳細プロセスは [product/README.md](./product/README.md#優先度付けプロセス) を参照。

---

## 第4条 機能単位のドキュメント化

新規機能を開発・改善するときは、`docs/` に **機能単位の仕様ファイル** を作る。
- 仕様の置き場: [product/features/](./product/features/)
- 実装メモ・ADR: [engineering/](./engineering/)
- AI 機能であれば: [ai/features/](./ai/features/)

すべての機能は次の三つ組をマッピングしなければならない。
1. **プロダクト方向性**（第1条のフライホイールにどう寄与するか）
2. **優先度**（第3条の受注 / チャーン回避）
3. **定量評価指標**（第5条）

テンプレート: [product/features/_template.md](./product/features/_template.md)

---

## 第5条 定量評価指標

すべての項目は **数値として定量的に評価できる指標** まで落とし込む。
- ペルソナ UX の差分 → 離脱率・コンバージョン・NPS など
- 競合劣位の解消 → 機能カバレッジ・検索順位・取得ユーザ比率など
- AI 機能 → eval スコア・人手レビュー一致率・ハルシネーション率など

横断的な指標台帳は [metrics/README.md](./metrics/README.md) に置く。

---

## 第6条 Good / Bad データセット駆動の開発プロセス

新規機能・改善は次のサイクルで進める。

1. **データセット作成**
   - Good / Bad の事例をいくつか定義する
   - UI/UX テストであれば Playwright を通じた Good / Bad 挙動定義
2. **Output 定義**
   - 当該データセットに対する期待 output を定義する
3. **評価指標設計**
   - その output を **データサイエンス的に評価** する指標を設計する
4. **継続更新**
   - 開発と並行して、データ・検索・仕様書を **常に Update しながら** 作成していく

詳細は [engineering/testing.md](./engineering/testing.md)、[ai/evals/README.md](./ai/evals/README.md) を参照。

---

## 第7条 Memory としての docs/ の運用

docs/ は静的ドキュメント置き場ではなく、**動的に Update され続ける Memory** である。

- 学びを得たら **口頭知識を残さず必ずファイル化** する
- 既存ファイルを更新できる場合は新規作成しない
- 各ディレクトリの `README.md` をインデックスとして維持する
- 古くなった内容は速やかに修正・削除する（陳腐化した記述は誤判断の温床）

## 改正手続き

- 本憲法は議論の上で改正される。改正時は git history に **理由が残る** コミットメッセージを必須とする。
- 個別ファイルが本憲法と矛盾する場合、**本憲法が優先** される。矛盾を見つけたらどちらかを修正すること。
