# product/ — プロダクト戦略

[CONSTITUTION.md](../CONSTITUTION.md) で定めた「プロダクトが目指す方向性」「優先度付けの基準」を実務に落とすためのフォルダ。

## ファイル

- `README.md`（本ファイル）— 索引・ペルソナ・優先度プロセス・マッピング
- `features/` — 機能単位の仕様書

---

## ペルソナ

主要ペルソナは 3 種類。詳細プロファイルは機能ごとの仕様書で具体化する。

| # | ペルソナ | 主目的 | サッカーNote での価値 |
| --- | --- | --- | --- |
| P1 | ユース年代の選手 | 個人スキル・思考の言語化 | 自分の試合認知を蓄積し、課題を再発見する |
| P2 | アマチュア選手 | 仲間との戦術合意・共有 | 個人ノートをチーム横断で参照する |
| P3 | コーチ・監督 | チーム単位の戦術言語化と指導 | 選手のノートを横断検索し、組織知として還元する |

各ペルソナで **UX 差分**（離脱地点・体験ギャップ）を観察し、機能仕様書 (`features/<name>.md`) に記録する。

---

## 優先度付けプロセス

1. 提案された機能 / 改善が **受注（acquisition）** か **チャーン回避（retention）** のどちらに効くかを **必ずユーザに確認** する（憲法 第3.2条）
2. プロダクト方向性（フライホイール寄与）×優先度×定量指標の **三つ組をマッピング** する
3. マッピング結果を `features/<name>.md` の冒頭に明記する
4. マッピングが書けないものは「優先度判断不能」として保留する

---

## マッピングフォーマット

`features/_template.md` の冒頭セクションと一致させる。

```yaml
mapping:
  product_direction: "選手ノートの動的検索"   # 第1条フライホイールへの寄与
  priority_axis: "retention"                # acquisition | retention
  quantitative_metric:
    name: "週次アクティブノート編集率"
    baseline: 0.21
    target: 0.40
    measurement: "週次でユニーク編集ユーザ / WAU を集計"
```

このマッピングが書けない機能は CONSTITUTION と整合していないので着手しない。
